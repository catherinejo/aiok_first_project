## 3. 성능 테스트 계획

### 3.1 테스트 유형

| 유형 | 목적 | 도구 |
|------|------|------|
| 단일 요청 테스트 | 기본 응답 시간 측정 | curl, httpx |
| 부하 테스트 | 동시 사용자 처리 능력 | locust, k6 |
| 스트레스 테스트 | 한계점 확인 | locust |
| 지속성 테스트 | 장시간 안정성 | k6 |

### 3.2 테스트 시나리오

#### 시나리오 1: 기본 부하 테스트

```python
# locustfile.py
from locust import HttpUser, task, between

class AIOKUser(HttpUser):
    wait_time = between(1, 3)

    @task(3)
    def chat_simple(self):
        self.client.post("/api/v1/chat", json={
            "message": "안녕하세요",
            "user_id": "test_user"
        })

    @task(2)
    def chat_translate(self):
        self.client.post("/api/v1/chat", json={
            "message": "Hello를 한국어로 번역해줘",
            "user_id": "test_user"
        })

    @task(1)
    def health_check(self):
        self.client.get("/health")
```

#### 시나리오 2: 점진적 부하 증가

```
시간(분)  |  동시 사용자
---------+-------------
0-2      |  10
2-5      |  25
5-8      |  50
8-10     |  75 (스트레스)
```

### 3.3 테스트 환경

| 환경 | 스펙 | 용도 |
|------|------|------|
| Local | M1 Mac, 16GB | 개발 테스트 |
| Staging | GCP e2-medium | 부하 테스트 |
| Production | GCP n2-standard-2 | 실제 운영 |

## 4. 성능 최적화 전략

### 4.1 응답 시간 최적화

| 영역 | 현재 | 최적화 방안 | 기대 효과 |
|------|------|------------|----------|
| **LLM 호출** | 2-5초 | 스트리밍 응답 | 체감 속도 향상 |
| **세션 조회** | 10ms | 메모리 캐싱 | < 1ms |
| **MCP 연결** | 500ms | 커넥션 풀링 | < 100ms |
| **직렬화** | 5ms | orjson 사용 | < 1ms |

### 4.2 처리량 최적화

| 방안 | 설명 |
|------|------|
| **비동기 처리** | FastAPI + asyncio 활용 |
| **워커 증설** | uvicorn --workers N |
| **수평 확장** | GCP Cloud Run 오토스케일링 |
| **캐싱** | 동일 요청 캐싱 (번역 등) |

### 4.3 비용 최적화

| 방안 | 절감 효과 |
|------|----------|
| 프롬프트 최적화 | 토큰 20% 절감 |
| 캐싱 활용 | 중복 호출 30% 감소 |
| 모델 선택 | Flash vs Pro 상황별 사용 |
| Rate Limiting | 과도한 사용 방지 |
